# Lumen 系统架构设计

## 1. 项目定位

Lumen 是一款跨平台的个人 AI 智能助手应用。她以科研辅助为起点，逐步演进为一个长期陪伴、持续成长、真正了解用户的个人 AI 实体。

### 核心特性

- **跨平台单体应用**：Android APK / Desktop 安装包 / 自部署服务端，一套代码
- **离线/在线双模式**：离线模式单设备独立运行，在线模式多设备实时同步
- **用户自有数据**：API key 由用户自行提供，数据完全由用户掌控
- **面向非技术用户**：安装即用，无需服务器或技术背景

## 2. 运行模式

### 2.1 离线模式

应用内嵌完整引擎，所有数据存储在本地设备。

```
┌──────────────────────────┐
│  Lumen App               │
│  ┌────────────────────┐  │
│  │ shared/ (全部逻辑)  │  │
│  │ Koog Agent          │  │
│  │ ObjectBox (本地数据) │  │
│  │ 记忆层 (本地向量)    │  │
│  └────────────────────┘  │
│  Compose UI              │
│  ──→ 直接调 LLM API      │
└──────────────────────────┘
```

- 无需任何后端服务，用户安装后输入 API key 即可使用
- 设备间数据同步通过导出/导入 `.lumen` 存档文件实现（手动传输）

### 2.2 在线模式

用户自部署 Lumen Server，多个客户端连接至同一后端，数据实时同步。

```
┌───────────┐         ┌──────────────────┐
│ Lumen App │ ──API─→ │ Lumen Server     │
│ (轻客户端) │         │ (Ktor)           │
│ Compose UI│ ←────── │ shared/ 全部逻辑  │
└───────────┘         │ ObjectBox         │
                      │ + LangBot Bridge  │
      多个设备          │ + ntfy 推送       │
      同时连接          └──────────────────┘
```

- 适合有服务器的技术用户
- 支持 LangBot 消息桥接（QQ/Telegram 等）和 ntfy 推送

### 2.3 存档同步（离线模式）

离线用户如需在设备间迁移数据，通过存档文件实现：

```
设备 A                          设备 B
  │                               │
  ├─ 导出 .lumen 存档 ──→ 通过微信/网盘/USB 传输 ──→ 导入存档
  │  (SQLite + 向量数据 + 配置)                       │
  └───────────────────────────────────────────────────┘
```

## 3. 演进路线

### 第一阶段：研究助理

- RSS / arxiv 信息采集与聚合
- LLM 驱动的中文摘要、趋势分析、与用户研究方向的关联分析
- 每日 Digest 推送（ntfy / 应用内通知）
- 订阅源管理界面

### 第二阶段：感情助手 + 对话

- 多轮对话，可设定角色人格
- 基于记忆层的上下文感知对话
- 工具调用能力（搜索、查询文章库、分析讨论等）
- 情感陪伴功能

### 第三阶段：长期记忆 + 知识图谱

- 情景记忆：记住每次对话的关键内容
- 知识图谱：用户论文、观点、项目进展全部关联
- 偏好学习：了解用户的兴趣、习惯、作息
- 主动关联：跨时间跨主题的信息连接

### 第四阶段：自主 Agent

- 主动推荐与发现
- 独立执行复杂任务
- 跨工具协作（笔记、邮件、日程等）
- 工作流自动化

### 第五阶段：多模态 + 生活融入

- 语音交互
- 图像理解
- 生活感知（日程、健康、天气）
- 情绪感知

## 4. 分层架构

### 4.1 最终形态

```
感知层 (Perception)  ：RSS/arxiv、新闻、日程、传感器、语音
记忆层 (Memory)      ：短期记忆、情景记忆、知识图谱（SimpleMem 移植）
思考层 (Reasoning)   ：LLM 推理、工具调用、规划、情感模型（Koog Agent）
行动层 (Action)      ：对话回复、推送通知、执行任务、调用 API
人格层 (Persona)     ：性格设定、语言风格、情感状态、成长性
```

## 5. 与外部系统的关系

| 系统 | 关系 | 模式 |
|------|------|------|
| LLM API (OpenAI/Anthropic/DeepSeek 等) | Lumen 通过 Koog 直接调用，用户自带 API key | 离线 + 在线 |
| LangBot | Lumen Server 通过 Bridge Plugin 桥接消息平台 | 仅在线 |
| ntfy | Lumen Server 主动推送通知，手机端接收 | 仅在线 |
| One-API | 高级用户可选，作为 LLM 统一网关（填入 api_base） | 可选 |

### 5.1 LangBot 集成架构

Lumen 不依赖 LangBot。在线模式下，通过极轻量的 Bridge Plugin 实现消息桥接：

```
QQ / Telegram / 微信 ...
        ↓
LangBot
  └── Lumen Bridge Plugin
        ├── EventListener: 拦截消息 → HTTP 转发至 Lumen Server API
        └── 收到 Lumen 响应 → 回复到消息平台

Lumen Server
  └── /api/langbot/webhook  ← 接收消息
      处理（Agent + 记忆 + 人格）
      返回响应 → Bridge Plugin → 消息平台
```

- Bridge Plugin 不调用 LangBot 的 LLM，纯消息转发
- Lumen 拥有完整的对话控制权（prompt、记忆注入、人格设定）

## 6. 设计原则

1. **跨平台优先**：所有核心逻辑在 shared 模块中实现，平台层只做 UI 和系统适配
2. **离线优先**：离线模式是一等公民，在线模式是增强而非必须
3. **不重复造轮子**：优先使用现成框架和库，自己只写业务逻辑
4. **用户数据自主**：API key、对话记录、记忆数据完全由用户掌控
5. **渐进式复杂度**：对非技术用户提供开箱即用体验，对技术用户开放高级配置
6. **接口抽象仅用于确定会替换的组件**：不过度设计，不为假设性需求预留抽象
